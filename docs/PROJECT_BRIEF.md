# 프로젝트 기획서: 토큰증권(STO) 발행·유통 샌드박스

## 목적
가상 자산을 토큰화해 발행하고, 규제 컴플라이언스 로직(투자자 적격성·투자한도·전매제한)을 적용하며, 2차 시장에서 유통하는 토큰증권 플랫폼을 구현한다. 코스콤이 신사업으로 추진하는 토큰증권(STO) 공동플랫폼의 개념을 이해하고 핵심 원리를 직접 구현해보는 것이 목표다. 테스트넷(Sepolia) 기반의 교육·포트폴리오 데모이며, 실제 증권 발행·실자금과 무관하다.

## 차별화 포인트 (가장 중요)
단순히 토큰을 발행하는(ERC-20 찍어내는) 프로젝트는 흔하다. 이 프로젝트의 차별점은 "금융 규제를 코드로 구현했다"는 것이다 — 아무나 못 사고(투자자 적격성), 한도를 넘겨 못 사고(투자한도), 일정 기간 못 파는(전매제한) 규제 로직을 스마트컨트랙트 레벨에서 강제한다. 이것이 PB/세무사의 전유물이던 복잡한 규제(Finance)를 소프트웨어(Tech)로 풀어내는, 금융과 IT의 결합을 가장 잘 보여주는 지점이다.

## 전체 흐름
발행자(Issuer)가 자산을 토큰화 → 투자자 등록(적격성 심사) → 컴플라이언스 검증(한도·전매제한) → 발행(1차) → 2차 시장 유통(매칭)

## 기술 스택
- Solidity, Hardhat, OpenZeppelin
- Sepolia 테스트넷
- 백엔드: Node.js/TypeScript 또는 Python (ethers.js/web3.py)
- 프론트엔드: React + TypeScript + ethers.js (MetaMask 연동)
- 배포: 컨트랙트는 Sepolia, 프론트는 정적 호스팅(Vercel 등)

## 핵심 설계 결정 (먼저 합의할 것)

### 1. 토큰 표준 선택
ERC-20을 기반으로 하되 규제 기능을 확장한다. (실무에서는 ERC-1400/ERC-3643 같은 증권형 토큰 표준이 있는데, 학습 곡선을 고려해 ERC-20 + 커스텀 규제 로직으로 갈지, 표준을 그대로 쓸지는 Phase1에서 합의. OpenZeppelin 기반 권장.)

### 2. 컴플라이언스 엔진 (이 프로젝트의 핵심)
- 투자자 적격성: 등록된(화이트리스트) 투자자만 토큰을 받을 수 있다.
- 투자한도: 투자자 등급별로 보유 가능한 최대 수량 제한.
- 전매제한(lock-up): 발행 후 일정 기간 동안 전송 불가.
- 이 규칙들을 토큰 전송(transfer) 시점에 스마트컨트랙트가 강제로 검증한다.

### 3. 2차 시장
MINI-Exchange에서 구현한 가격-시간 우선 매칭 개념을 재사용한다. (온체인 매칭은 가스비가 비싸므로, 오프체인 매칭 + 온체인 정산 같은 현실적 구조를 택할지 Phase 단위로 판단.)

### 4. 보안
OpenZeppelin의 검증된 컨트랙트를 최대한 재사용하고, 재진입 공격·정수 오버플로우 등 알려진 취약점을 인지·문서화한다.

## 단계별 범위

### Phase 1 — MVP (먼저 끝낼 것)
- 가상 자산 토큰 발행 컨트랙트 (ERC-20 기반, OpenZeppelin)
- 기본 컴플라이언스: 투자자 화이트리스트(등록된 투자자만 토큰 수령 가능)
- Hardhat 테스트(발행, 전송 허용/거부 케이스)
- Sepolia 테스트넷 배포

### Phase 2 — 차별화 기능 (Phase1 완료 후)
- 투자한도 로직(등급별 최대 보유량)
- 전매제한(lock-up) 로직(발행 후 N일간 전송 불가)
- 발행자/투자자 등록 흐름 정교화
- 프론트엔드(MetaMask 연동) — 발행·투자·전송 UI
- 컴플라이언스 위반 시 트랜잭션이 거부되는 장면을 데모로 시연

### Phase 3 — 스트레치 (시간 남으면)
- 2차 시장(유통) — MINI-Exchange 매칭 개념 재사용
- 배당/이자 분배 같은 증권 라이프사이클 이벤트
- 발행 현황·거래 내역 대시보드

## 평가/검증 방법
- Hardhat 테스트: 컴플라이언스 규칙이 실제로 강제되는지(미등록 투자자 전송 거부, 한도 초과 거부, lock-up 기간 내 전송 거부) 케이스별 검증
- Sepolia 배포 후 실제 트랜잭션으로 발행·전송·거부 동작 확인
- Etherscan(Sepolia)에서 배포된 컨트랙트와 트랜잭션 확인 가능하게

## 주의사항 (법적/윤리)
- 테스트넷 기반 샌드박스/데모. 실제 증권 발행·실자금·실제 투자자 판매가 아님을 명시.
- 실제 STO는 자본시장법 등 규제 대상임을 명시.
- "코스콤 토큰증권 플랫폼을 그대로 재현"이 아니라 "개념을 이해하고 핵심 원리를 구현"으로 서술.
- 개인키/니모닉을 절대 깃에 커밋하지 않는다(.env + .gitignore).

## 포트폴리오 활용
- 코스콤 토큰증권 공동플랫폼 신사업과 직접 연결되는 스토리. "코스콤이 추진하는 STO를 보고 착안했다"고 설명 가능.
- 면접 대비: "컴플라이언스(적격성/한도/전매제한)를 어떻게 코드로 강제했는지", "왜 이 토큰 표준을 선택했는지", "스마트컨트랙트 보안을 어떻게 고려했는지", "2차 시장 매칭을 MINI-Exchange에서 어떻게 재사용했는지" 질문에 답할 수 있도록 설계 과정을 기록.
- MINI-Exchange와의 연결: "거래소 매칭엔진(MINI-Exchange) → 토큰증권 발행·유통(이 프로젝트)"으로 자본시장 인프라 이해의 폭을 보여줌.
